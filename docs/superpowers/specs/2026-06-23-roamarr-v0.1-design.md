# Roamarr v0.1 — Walking Skeleton Design

**Status:** approved scope, design draft (revised after adversarial peer review — see §12)
**Date:** 2026-06-23
**Scope:** A thin end-to-end vertical slice of a self-hosted, free TripIt/TripIt-Pro alternative. Every planned subsystem is present but deliberately shallow, proving the architecture and the seams between pieces. Later specs deepen each subsystem.

---

## 1. Goals & non-goals

### Goals (v0.1)
- A single self-hosted container a user runs on their own box, *arr-style.
- First-run admin onboarding; optional public self-registration; per-user data isolation.
- Manually add/edit trips and their segments (flights, lodging).
- A generic reminder engine driving flight check-in reminders and travel-document expiry reminders, delivered in-app and (optionally) by SMTP email.
- Travel profile: documents (passport / driver's license / Global Entry) and loyalty programs.
- Groups, trip sharing (with users/groups, read-only), and public token share links.
- Credit cards with reusable benefit templates, attachable to bookings.
- Travel-insurance policies attachable to trips.
- A **pluggable** fare-watch provider framework with one stub provider and per-user, multi-provider API keys.

### Non-goals (explicitly deferred or never)
- **Email auto-import** of confirmation emails — deferred to a later phase; architecture must leave room.
- **Real fare-watch / seat-watch logic** — never built ourselves; only third-party providers via the plug-in framework.
- Notification channels beyond in-app + SMTP (Discord/Telegram/ntfy/webhook) — later phase.
- Segment types beyond flight + lodging (car, rail, activity, cruise) — later.
- Richer shared-trip views (shared viewers seeing confirmation numbers etc.), per-user reminder leads, multiple accounts per fare provider, global benefit-template library, OAuth/SSO, 2FA, .ics calendar sync, mobile app, i18n, currency conversion, fare-drop/seat alerts — out of scope for v0.1.

> **Ponytail stance:** each subsystem stays minimal until the skeleton proves the shape. Depth is a later spec, not v0.1. Where this doc names a fixed value (reminder lead times, session length, argon2 params), it's a constant with a noted upgrade path, not a config surface yet. Security correctness is *not* a place we cut — see §8.

## 2. Tech foundation

| Concern | Choice | Why |
|---|---|---|
| Framework | **SvelteKit** + `@sveltejs/adapter-node` | Full-stack TS in one artifact, SSR + form actions, runs as a long-lived Node server (needed for the scheduler). |
| DB | **SQLite** via **Drizzle ORM** (`better-sqlite3`), WAL mode, `foreign_keys=ON` | Typed schema + migrations, single file, zero external service. |
| Auth | Hand-rolled server-side sessions + `@node-rs/argon2` | No heavy/abandoned dep; cookie holds a random token, DB stores only its hash (§8). |
| Scheduler | In-process 60-second tick loop, single-start guarded, non-overlapping | Single process, single container — a cron-syntax dep buys nothing. Atomically claims due reminders each minute (§6). |
| Styling | Tailwind CSS | Fast path to a polished UI. |
| At-rest crypto | `node:crypto` AES-256-GCM, versioned, random nonce, key from `ROAMARR_SECRET` (§8) | Encrypts the few genuinely sensitive fields. |
| Container | One multi-stage Dockerfile; SQLite on a mounted volume | Standard self-hosted distribution. |

DB client sets `PRAGMA journal_mode=WAL` and `PRAGMA foreign_keys=ON` on every connection. Migrations run on boot, before the scheduler starts.

## 3. Module boundaries (`src/lib/server/`)

Each module has one purpose, a small public surface, and is unit-testable in isolation.

- **`db/`** — Drizzle schema, migrations, and the singleton client (sets WAL + `foreign_keys=ON`). The only place that touches SQLite.
- **`auth.ts`** — `hashPassword`, `verifyPassword`, `createSession` (returns raw token; stores only its hash), `validateSession`, `invalidateSession`, `purgeExpiredSessions`, and `requireUser` / `requireAdmin` guards. Depends on `db`.
- **`settings.ts`** — load/save the single-row global config (`setup_complete`, `allow_registration`, SMTP, instance name, default timezone). Depends on `db`.
- **`ownership.ts`** — the IDOR guardrail. `requireOwnedTrip(user, tripId)`, `requireOwnedRow(table, id, user)`, and `assertOwnedRefs(user, {cardId?, tripId?, providerId?, segmentId?})` used by every mutation that takes a foreign id from the client. Returns the row or throws 403/404. Depends on `db`.
- **`sharing.ts`** — trip read authorization for logged-in users only. `canView(user, trip)` = owner ∨ direct share ∨ member of a group the trip is shared with. `canEdit(user, trip)` = owner only (shares are read-only in v0.1). **Does not consider public tokens** — public access is solely the `/share/[token]` route. `viewerProjection(trip)` = the itinerary-only view shown to every non-owner (shared *and* public), see §7. Depends on `db`.
- **`reminders.ts`** — `upsertRemindersForSegment(segment)`, `upsertRemindersForDocument(doc)`, `cancelRemindersFor(refType, refId)`, `computeFireAt(kind, instant, tz)` (pure, exported for tests), and `runDueReminders(now)` (the tick body, §6). Depends on `db`, `notify`.
- **`notify.ts`** — `deliver(userId, {title, body, link})`: always writes an in-app `notifications` row; sends SMTP if configured. Bodies never contain encrypted-class or private-class fields (§8). A `Channel` interface (`inApp`, `smtp`) so more channels plug in later. Depends on `db`, `settings`.
- **`crypto.ts`** — `encrypt(plain)` / `decrypt(blob)` (AES-256-GCM, versioned, §8). Depends on env only.
- **`fareproviders/`** — `interface FareProvider { key; label; check(watch, apiKey): Promise<FareResult> }`, a `registry` (`Record<key, FareProvider>`), and `stub.ts`. `runFareChecks(now)` is called from the tick. Adding a real provider = one new file + one registry entry.
- **`scheduler.ts`** — starts the single tick loop (§6). Imported once from `hooks.server.ts`; guarded by a `globalThis` flag against HMR/double-start; runs only in the Node server runtime, only after migrations.

`src/hooks.server.ts` resolves the session cookie into `event.locals.user`, redirects to `/setup` when `!setup_complete`, enforces auth on protected routes, and triggers `scheduler.ts` start once.

## 4. Data model (SQLite / Drizzle)

Datetimes are stored as ISO-8601 UTC **instants** (text). Dates (no time) are stored as `YYYY-MM-DD`. Every enum column gets a `CHECK`; every column its `NOT NULL`/default; `foreign_keys=ON` is enforced. `*_json` columns are display-only metadata — nothing behavioral (reminders, authorization, projection) ever reads them.

```
users(id pk, email unique /* normalized: trim+lowercase */, password_hash, display_name,
      role check in('admin','user'), timezone default 'UTC' /* IANA */, created_at)
sessions(id pk, token_hash unique /* sha-256 of cookie token; raw token never stored */,
         user_id fk users on delete cascade, expires_at, created_at)
settings(id pk =1, instance_name, setup_complete bool default 0, allow_registration bool default 0,
         default_timezone default 'UTC',
         smtp_host, smtp_port, smtp_user, smtp_pass /* encrypted */, smtp_from)

trips(id pk, owner_id fk users on delete cascade, name, destination, start_date, end_date, notes,
      default_visibility check in('private','groups','public') default 'private',  -- UI default only (§7)
      public_token unique nullable, created_at, updated_at)

segments(id pk, trip_id fk trips on delete cascade, type check in('flight','lodging'), title,
         start_at /* UTC instant */, start_tz /* IANA, departure-local */, end_at nullable,
         location, confirmation_number nullable,
         details_json /* DISPLAY-ONLY: airline, flight_no, dep/arr airport, seat, room... */,
         card_id fk cards on delete set null nullable, created_at, updated_at)

travel_documents(id pk, user_id fk users on delete cascade,
                 type check in('passport','drivers_license','global_entry'),
                 number /* encrypted */, issuing_authority, expires_on /* date */, notes)
loyalty_programs(id pk, user_id fk users on delete cascade,
                 program_name, membership_number, balance int nullable, notes)

groups(id pk, owner_id fk users on delete cascade, name, created_at)
group_members(group_id fk groups on delete cascade, user_id fk users on delete cascade,
              primary key(group_id, user_id))
trip_shares(id pk, trip_id fk trips on delete cascade,
            shared_with_user_id fk users on delete cascade nullable,
            shared_with_group_id fk groups on delete cascade nullable,
            permission check in('read') default 'read', created_at,
            check( (shared_with_user_id is not null) <> (shared_with_group_id is not null) ))
  -- + unique(trip_id, shared_with_user_id) where user not null
  -- + unique(trip_id, shared_with_group_id) where group not null

cards(id pk, user_id fk users on delete cascade,
      nickname, network check in('visa','mc','amex','disc','other'), last4, notes)
  -- NEVER store full PAN. last4 only. Sidesteps PCI entirely.
card_benefits(id pk, card_id fk cards on delete cascade,
              benefit_type check in('trip_delay','baggage_delay','trip_cancellation','other'),
              coverage_amount int nullable, currency default 'USD', notes)

insurance_policies(id pk, user_id fk users on delete cascade, provider, policy_number,
                   coverage_summary, coverage_amount int nullable, currency default 'USD',
                   start_date, end_date, trip_id fk trips on delete set null nullable, notes)

fare_providers(id pk, user_id fk users on delete cascade, provider_key, api_key /* encrypted */,
               enabled bool default 1, config_json, unique(user_id, provider_key))
fare_watches(id pk, trip_id fk trips on delete cascade, segment_id fk segments on delete cascade nullable,
             provider_id fk fare_providers on delete cascade,
             status check in('active','paused') default 'active',
             last_checked_at nullable, last_result_json, created_at)

reminders(id pk, user_id fk users on delete cascade,
          kind check in('flight_checkin','document_expiry'),
          ref_type check in('segment','document'), ref_id, fire_at,
          status check in('pending','sending','sent') default 'pending',
          attempts int default 0, sent_at nullable, created_at,
          unique(kind, ref_type, ref_id))
  -- ref is polymorphic (no FK): source deletes call cancelRemindersFor() explicitly.
notifications(id pk, user_id fk users on delete cascade, title, body, link nullable,
              created_at, read_at nullable)
```

**Indexes:** `trips(owner_id)`, `trips(start_date)`, `segments(trip_id)`, `segments(start_at)`, `trip_shares(shared_with_user_id)`, `trip_shares(shared_with_group_id)`, `group_members(user_id)`, `reminders(status, fire_at)`, `notifications(user_id, read_at)`, `fare_watches(provider_id)`, `fare_watches(status)`, `travel_documents(user_id, expires_on)`, `insurance_policies(trip_id)`, `cards(user_id)`, `loyalty_programs(user_id)`.

### Reminder generation & idempotency
- **Flight check-in:** on flight segment create/update, `computeFireAt('flight_checkin', start_at)` = `start_at − 24h` (a UTC instant; `start_at` is already a correct instant derived from local time + `start_tz` at entry).
- **Document expiry:** `computeFireAt('document_expiry', expires_on, user.timezone)` = 09:00 user-local on `expires_on − 90 days`, converted to UTC.
- **Idempotent upsert** keyed on `unique(kind, ref_type, ref_id)`: recompute `fire_at`; if no row, insert; if a row exists, update `fire_at` and **re-arm** to `status='pending'` whenever the computed `fire_at` differs and is in the future (so a corrected/rescheduled flight re-notifies even if the old reminder was already `sent`). If the new `fire_at` is already past at write time, set `status='sent'` (no late spam).
- **Cancellation:** segment/document delete, or a flight→lodging type change, calls `cancelRemindersFor(refType, refId)` which deletes the reminder row.

> The 24h / 90d leads and the 09:00 expiry hour are constants in `reminders.ts`. **Upgrade path:** per-user configurable leads later.

## 5. Routes & screens

Protected unless noted. SvelteKit `load` + form `actions`; progressive enhancement. Every action that accepts a foreign id runs it through `ownership.ts` first.

| Route | Purpose |
|---|---|
| `/setup` *(only when `!setup_complete`)* | First-run admin creation, **atomic** (§6). Sets instance name + timezone, flips `setup_complete`. |
| `/login`, `/logout` | Session login/logout. |
| `/register` *(404 unless `setup_complete && allow_registration`)* | Public self-registration; normalized email; always `role='user'`. |
| `/` | Dashboard: upcoming trips, unread notifications, soon-to-expire documents. |
| `/trips`, `/trips/new`, `/trips/[id]`, `/trips/[id]/edit` | Trip list / create / **owner** detail (timeline, shares, attached cards & insurance, fare-watch toggles) / edit. Non-owner viewers get `viewerProjection` (§7), not this detail. |
| `/trips/[id]/segments` *(actions)* | Add/edit/delete flight & lodging segments; attach a card (ownership-checked). |
| `/profile/documents`, `/profile/loyalty` | Travel documents (with expiry) and loyalty programs. |
| `/cards`, `/insurance` | Cards + per-card benefit templates; insurance policies. |
| `/groups` | Create groups, add/remove members. |
| `/notifications` | In-app inbox; mark read/unread (`read_at`). *(No separate dismiss in v0.1.)* |
| `/settings` *(admin)* | `allow_registration`, SMTP config, instance name, default timezone. |
| `/settings/fare-providers` *(per-user)* | Enable providers, enter/rotate API key(s); one row per `provider_key`. |
| `/share/[token]` *(public, no auth)* | Token lookup → `viewerProjection` only. No `canView` call. |

## 6. Key data flows

**Atomic first-run setup.** `/setup` POST runs in one `BEGIN IMMEDIATE` transaction: re-check `setup_complete=0` **and** `users` count = 0; if either fails, abort → redirect to `/login`. Otherwise insert exactly one `role='admin'` user and set `setup_complete=1`. SQLite write serialization + the in-transaction re-check closes the parallel-request / late-visitor takeover window.

**Reminder lifecycle.** Create flight → segment action writes the row → `upsertRemindersForSegment` arms a `pending` reminder at `start_at−24h`. The tick **atomically claims** due rows: `UPDATE reminders SET status='sending' WHERE status='pending' AND fire_at<=:now RETURNING *`. For each claimed row it calls `notify.deliver`; on success → `status='sent', sent_at=now`; on failure → back to `pending`, `attempts+1` (after N attempts, give up and mark `sent` with a logged error). The claim-before-deliver flip makes delivery idempotent across crashes, tick overlap, or an accidental second scheduler.

**Scheduler start.** `hooks.server.ts` calls `scheduler.start()` once; a `globalThis.__roamarr_scheduler` flag prevents HMR/import double-start; it runs only in the Node server runtime, only after migrations. Each tick sets a `running` flag and skips if the previous tick hasn't finished (no overlap).

**Authorization.** Every authenticated trip route loads the trip then calls `sharing.canView`/`canEdit`. The owner sees full detail; any other viewer (shared) sees `viewerProjection`. Every mutation taking a client-supplied foreign id (card, trip, provider, segment, group) passes through `ownership.ts`, which joins back to the current user and 403/404s on mismatch. `/share/[token]` bypasses login, does a token lookup, and returns only `viewerProjection`.

**Fare-watch.** User enables a provider in `/settings/fare-providers` (stores encrypted key) → on a trip/flight, toggling "fare watch via [provider]" creates a `fare_watch` (provider ownership-checked) → the tick calls `provider.check(watch, apiKey)` for `active` watches of `enabled` providers and stores `last_result_json` (the stub returns a placeholder; we never compute fares).

## 7. Trip viewing & sharing model

- **Owner** (`canEdit`): full detail — segments, confirmation numbers, attached cards/benefits, insurance, fare-watch, shares, edit.
- **Authenticated shared viewer** (`canView`, not owner) **and public token viewer**: the *same* `viewerProjection`. It returns **only** instance name, trip name, destination, dates, and segments (type, title, times, location). It **excludes** `notes`, `confirmation_number`, cards & benefits, insurance, travel documents, loyalty, owner email, shares, and fare-watch data.
- `default_visibility` is a **UI default only**: it pre-selects the create-form sharing controls and, when `public`, mints a `public_token` at creation. It is **not** an access-control field — access is governed entirely by `trip_shares` + `public_token`.
- Public links are read-only; revoking = clearing `public_token`.

> **Ponytail simplification:** v0.1 gives shared viewers the same minimal projection as public links. Richer shared views (e.g. confirmation numbers for family members) are a later "share level" feature — marked, not built.

## 8. Security & sensitive data

- **Passwords:** argon2id, pinned params (OWASP floor): memory 19 MiB (`m=19456`), iterations `t=2`, parallelism `p=1`, 16-byte random salt (library), reject passwords < 8 or > 1024 bytes.
- **Sessions:** cookie holds a 32-byte random URL-safe token; the DB stores only `sha-256(token)` (`token_hash`). Cookie `httpOnly`, `secure`, `sameSite=lax`; 30-day expiry; validated server-side every request (lookup by hash); invalidated on logout; expired rows purged periodically. A stolen DB copy yields no live sessions.
- **At-rest encryption** (`crypto.ts`, AES-256-GCM): `ROAMARR_SECRET` is decoded as base64; if it is not exactly 32 bytes it is run through `scrypt` with a fixed app salt to a 32-byte key. `encrypt()` uses a fresh random 96-bit nonce per call and emits `v1.<nonce-b64>.<ciphertext+tag-b64>`. `decrypt()` parses the version, splits, and lets GCM auth verify (tamper → throw). App refuses to boot without `ROAMARR_SECRET`. **Encrypted fields:** `travel_documents.number`, `fare_providers.api_key`, `settings.smtp_pass`.
- **Data classification** (response to "encrypt everything"): true secrets and government-ID numbers are encrypted (above). Booking identifiers — `confirmation_number`, `membership_number`, `policy_number`, and all `notes` — are **plaintext-but-private**: not credentials, so they stay queryable/displayable, but they are owner-only and **must never appear in notifications, email bodies, or any `viewerProjection`**. Card data is `last4 + network` only — never a full PAN, removing PCI scope by construction.
- **Per-user isolation:** centralized in `ownership.ts`. Every mutation accepting a client foreign id verifies it joins back to the current user before use; cross-owner ids 403/404. Read paths scope by `owner_id`/`user_id` or go through `sharing.canView`.
- **Registration:** disabled by default; `/register` 404s until setup is complete and an admin enables it; registrants are always `role='user'`.

## 9. Testing (v0.1)

Vitest, focused on the security- and logic-critical seams (ponytail: one runnable check per non-trivial path). Unit + a thin layer of route/action integration tests where the real boundary lives.

**Unit:**
1. `sharing.canView/canEdit` — owner, shared-user, group-member, stranger (public token is *not* a canView path).
2. `reminders.computeFireAt` + upsert — fire times (incl. timezone conversion), idempotent re-upsert, re-arm on changed future time, past-time → immediate `sent`, cancel on delete.
3. `auth` — argon2id hash/verify round-trip; session validate by token-hash + expiry; raw token never persisted.
4. `crypto` — encrypt/decrypt round-trip; **distinct nonces across calls**; tampered blob rejected.
5. `fareproviders.registry` — resolves the stub; `runFareChecks` skips paused watches / disabled providers.

**Integration (the actual attack surface):**
6. Cross-owner IDOR — user B cannot read/mutate user A's trip, segment, card, insurance, or fare-watch by id.
7. Projection leakage — `viewerProjection` (shared *and* `/share/[token]`) omits notes, confirmation numbers, cards, insurance, documents, loyalty, fare-watch.
8. Setup race — a second `/setup` POST after completion is rejected; no second admin.
9. Session cookie attributes (`httpOnly`/`secure`/`sameSite`) and atomic reminder claim (no double-deliver).

## 12. Peer-review revisions (2026-06-23)

Adversarial review (codex, read-only) raised 2 critical / 11 high / 9 medium. Incorporated:

- **Atomic first-run setup** (§6) — closes the parallel/late-visitor admin-takeover hole.
- **Per-user isolation via `ownership.ts`** (§3/§8) — every client foreign id is owner-checked; IDOR integration tests added (§9).
- **Sessions store a token hash, not the raw token** (§4/§8).
- **AES-GCM fully specified** — key derivation, random 96-bit nonce, versioned `v1.nonce.ct+tag`, tamper test (§8).
- **argon2id params pinned** (§8).
- **`foreign_keys=ON` + CHECK constraints + indexes + cascade/SET NULL rules** (§4).
- **Timezones** — `segments.start_tz`, `users.timezone`; check-in and expiry fire-times computed correctly (§4).
- **Idempotent reminder re-arm** on changed times; **atomic claim** delivery to prevent duplicates (§4/§6).
- **Shared viewers no longer leak** owner cards/insurance — one `viewerProjection` for shared + public; `notes` excluded; `canView` no longer touches public tokens; `default_visibility` demoted to a UI default (§3/§7).
- **Registration hardening** — normalized email, `role='user'`, gated on setup (§5/§8).
- **Schema cleanups** — `trip_shares` exactly-one CHECK + unique partial indexes, `unique(user_id, provider_key)`, `depart_at`→`start_at`, `details_json` declared display-only, dismiss dropped (read/unread only), reminder statuses trimmed to `pending|sending|sent` (§4/§5).

**Rejected as over-engineering:** encrypting `confirmation_number` / `membership_number` / `policy_number` / `notes`. These are booking identifiers, not credentials; encrypting them costs queryability/display for marginal gain over the DB-theft threat the true secrets already cover. Handled by **data classification** instead (§8): encrypt secrets + government IDs, keep the rest plaintext-but-private with a no-leak rule.
