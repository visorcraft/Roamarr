# Roamarr v0.1 — Walking Skeleton Design

**Status:** approved scope, design draft
**Date:** 2026-06-23
**Scope:** A thin end-to-end vertical slice of a self-hosted, free TripIt/TripIt-Pro alternative. Every planned subsystem is present but deliberately shallow, proving the architecture and the seams between pieces. Later specs deepen each subsystem.

---

## 1. Goals & non-goals

### Goals (v0.1)
- A single self-hosted container a user runs on their own box, *arr-style.
- First-run admin onboarding; optional public self-registration; per-user data isolation.
- Manually add/edit trips and their segments (flights, lodging).
- A generic reminder engine driving flight check-in reminders and travel-document expiry reminders, delivered in-app and (optionally) by SMTP email.
- Travel profile: documents (passport / driver's license / Global Entry) and loyalty programs.
- Groups, trip sharing (with users/groups, read-only), and public token share links.
- Credit cards with reusable benefit templates, attachable to bookings.
- Travel-insurance policies attachable to trips.
- A **pluggable** fare-watch provider framework with one stub provider and per-user, multi-provider API keys.

### Non-goals (explicitly deferred or never)
- **Email auto-import** of confirmation emails — deferred to a later phase; architecture must leave room.
- **Real fare-watch / seat-watch logic** — never built ourselves; only third-party providers via the plug-in framework.
- Notification channels beyond in-app + SMTP (Discord/Telegram/ntfy/webhook) — later phase.
- Segment types beyond flight + lodging (car, rail, activity, cruise) — later.
- Global/shared benefit-template library, OAuth/SSO, 2FA, .ics calendar sync, mobile app, i18n, currency conversion, fare-drop/seat alerts — out of scope for v0.1.

> **Ponytail stance:** each subsystem stays minimal until the skeleton proves the shape. Depth is a later spec, not v0.1. Where this doc names a fixed value (reminder lead times, session length), it's a constant with a noted upgrade path, not a config surface yet.

## 2. Tech foundation

| Concern | Choice | Why |
|---|---|---|
| Framework | **SvelteKit** + `@sveltejs/adapter-node` | Full-stack TS in one artifact, SSR + form actions, runs as a long-lived Node server (needed for the scheduler). |
| DB | **SQLite** via **Drizzle ORM** (`better-sqlite3`) | Typed schema + migrations, single file, zero external service. |
| Auth | Hand-rolled server-side sessions + `@node-rs/argon2` | No heavy/abandoned dep; sessions are a token cookie + a `sessions` row. |
| Scheduler | In-process 60-second tick loop (`setInterval`) | Single process, single container — a cron-syntax dep buys nothing. Scans due reminders each minute. |
| Styling | Tailwind CSS | Fast path to a polished UI. |
| At-rest crypto | `node:crypto` AES-256-GCM, key from `ROAMARR_SECRET` | Encrypts the few genuinely sensitive fields (see §8). |
| Container | One multi-stage Dockerfile; SQLite on a mounted volume | Standard self-hosted distribution. |

Migrations run on boot. The scheduler starts once when the server boots (guarded so SvelteKit's dev/prod single instance starts exactly one loop).

## 3. Module boundaries (`src/lib/server/`)

Each module has one purpose, a small public surface, and is unit-testable in isolation.

- **`db/`** — Drizzle schema, migrations, and the singleton client. The only place that touches SQLite.
- **`auth.ts`** — `hashPassword`, `verifyPassword`, `createSession`, `validateSession`, `invalidateSession`, and `requireUser` / `requireAdmin` guards. Depends on `db`.
- **`settings.ts`** — load/save the single-row global config (`setup_complete`, `allow_registration`, SMTP, instance name). Depends on `db`.
- **`sharing.ts`** — **the one authorization choke point for trips.** `canView(user, trip)` and `canEdit(user, trip)` resolve owner / shared-user / group-member / public-token / stranger. Also `publicTripProjection(trip)` (the safe read-only view). Depends on `db`.
- **`reminders.ts`** — `upsertRemindersForSegment(segment)`, `upsertRemindersForDocument(doc)`, `cancelRemindersFor(refType, refId)`, and `runDueReminders(now)` (the tick body). Pure-ish: `computeFireAt` is a separate exported function for testing. Depends on `db`, `notify`.
- **`notify.ts`** — `deliver(userId, {title, body, link})`: always writes an in-app `notifications` row; sends SMTP if configured. A `Channel` interface (`inApp`, `smtp`) so more channels plug in later. Depends on `db`, `settings`.
- **`crypto.ts`** — `encrypt(plain)` / `decrypt(blob)` (AES-256-GCM) for sensitive fields. Depends on env only.
- **`fareproviders/`** — `interface FareProvider { key; label; check(watch, apiKey): Promise<FareResult> }`, a `registry` (`Record<key, FareProvider>`), and `stub.ts`. `runFareChecks(now)` is called from the tick. Adding a real provider = one new file + one registry entry.

`src/hooks.server.ts` resolves the session cookie into `event.locals.user`, redirects to `/setup` when `!setup_complete`, and enforces auth on protected routes.

## 4. Data model (SQLite / Drizzle)

Datetimes stored as ISO-8601 text (UTC). `*_json` columns are text holding JSON. All `user_id` FKs scope data to an owner.

```
users(id pk, email unique, password_hash, display_name, role['admin'|'user'], created_at)
sessions(id pk /* random token */, user_id fk, expires_at)
settings(id pk =1, instance_name, setup_complete bool, allow_registration bool,
         smtp_host, smtp_port, smtp_user, smtp_pass /* encrypted */, smtp_from)

trips(id pk, owner_id fk users, name, destination, start_date, end_date, notes,
      default_visibility['private'|'groups'|'public'] default 'private',
      public_token unique nullable, created_at, updated_at)

segments(id pk, trip_id fk, type['flight'|'lodging'], title,
         start_at, end_at nullable, location, confirmation_number nullable,
         details_json /* type-specific: airline, flight_no, dep/arr airport, seat, room... */,
         card_id fk cards nullable, created_at, updated_at)

travel_documents(id pk, user_id fk, type['passport'|'drivers_license'|'global_entry'],
                 number /* encrypted */, issuing_authority, expires_on date, notes)
loyalty_programs(id pk, user_id fk, program_name, membership_number, balance int nullable, notes)

groups(id pk, owner_id fk users, name, created_at)
group_members(group_id fk, user_id fk, primary key(group_id, user_id))
trip_shares(id pk, trip_id fk, shared_with_user_id fk nullable, shared_with_group_id fk nullable,
            permission['read'] default 'read', created_at)
  -- exactly one of shared_with_user_id / shared_with_group_id is set

cards(id pk, user_id fk, nickname, network['visa'|'mc'|'amex'|'disc'|'other'], last4, notes)
  -- NEVER store full PAN. last4 only. Sidesteps PCI entirely.
card_benefits(id pk, card_id fk, benefit_type['trip_delay'|'baggage_delay'|'trip_cancellation'|'other'],
              coverage_amount int nullable, currency default 'USD', notes)

insurance_policies(id pk, user_id fk, provider, policy_number, coverage_summary,
                   coverage_amount int nullable, currency default 'USD',
                   start_date, end_date, trip_id fk nullable, notes)

fare_providers(id pk, user_id fk, provider_key, api_key /* encrypted */, enabled bool, config_json)
  -- multiple rows per user => multiple providers, each with its own key
fare_watches(id pk, trip_id fk, segment_id fk nullable, provider_id fk fare_providers,
             status['active'|'paused'], last_checked_at nullable, last_result_json, created_at)

reminders(id pk, user_id fk, kind['flight_checkin'|'document_expiry'],
          ref_type['segment'|'document'], ref_id, fire_at,
          status['pending'|'sent'|'dismissed'|'cancelled'], created_at)
  -- unique(kind, ref_type, ref_id) so generation is an idempotent upsert
notifications(id pk, user_id fk, title, body, link nullable, created_at, read_at nullable)
```

### Reminder generation rules
- **Flight check-in:** on flight segment create/update, upsert a `flight_checkin` reminder at `depart_at − 24h`. If that time is already past at creation, mark `sent` immediately (no late spam). Delete/cancel on segment delete or type change.
- **Document expiry:** on document create/update with an `expires_on`, upsert a `document_expiry` reminder at `expires_on − 90 days`. (Single lead for v0.1.)

> `computeFireAt(kind, refDate)` and the 24h / 90d leads are constants in `reminders.ts`. **Upgrade path:** move leads to per-user settings when users ask for configurable lead times.

## 5. Routes & screens

Protected unless noted. SvelteKit `load` + form `actions`; progressive enhancement.

| Route | Purpose |
|---|---|
| `/setup` *(only when `!setup_complete`)* | First-run wizard: create admin, set instance name. Flips `setup_complete`. |
| `/login`, `/logout` | Session login/logout. |
| `/register` *(404/disabled when `!allow_registration`)* | Public self-registration. |
| `/` | Dashboard: upcoming trips, unread notifications, soon-to-expire documents. |
| `/trips`, `/trips/new`, `/trips/[id]`, `/trips/[id]/edit` | Trip list / create / detail (timeline of segments, shares, attached cards & insurance, fare-watch toggles) / edit. |
| `/trips/[id]/segments` *(actions)* | Add/edit/delete flight & lodging segments; attach a card. |
| `/profile/documents`, `/profile/loyalty` | Travel documents (with expiry) and loyalty programs. |
| `/cards`, `/insurance` | Cards + per-card benefit templates; insurance policies. |
| `/groups` | Create groups, add/remove members. |
| `/notifications` | In-app inbox; mark read / dismiss. |
| `/settings` *(admin)* | `allow_registration`, SMTP config, instance name. |
| `/settings/fare-providers` *(per-user)* | Enable providers, enter/rotate API key(s). |
| `/share/[token]` *(public, no auth)* | Read-only trip view via `publicTripProjection`. |

## 6. Key data flows

**Reminder lifecycle.** Create flight → segment action writes the row → `upsertRemindersForSegment` creates a pending reminder at `depart−24h`. Every 60s `runDueReminders(now)` selects `status='pending' AND fire_at<=now`, calls `notify.deliver` (in-app row + SMTP if configured), marks `sent`. Dashboard/inbox read `notifications`.

**Authorization.** Every trip route loads the trip then calls `sharing.canView`/`canEdit`. `/share/[token]` bypasses login but returns only `publicTripProjection`. No trip access exists outside `sharing.ts`.

**Fare-watch.** User enables a provider in `/settings/fare-providers` (stores encrypted key) → on a trip/flight, toggling "fare watch via [provider]" creates a `fare_watch` → the tick calls `provider.check(watch, apiKey)` and stores `last_result_json` (the stub returns a placeholder result; we never compute fares). Disabled providers are skipped.

---

## 7. Public share projection (§ what a token link exposes)

`publicTripProjection(trip)` returns **only**: instance name, trip name, destination, dates, notes, and segments (type, title, times, location). It **excludes**: confirmation numbers, attached cards & benefits, insurance, travel documents, loyalty, owner email, shares, and fare-watch data. Public links are read-only; revoking = clearing `public_token`.

---

## 8. Security & sensitive data

- **Passwords:** argon2id.
- **Sessions:** random 32-byte token; cookie `httpOnly`, `secure`, `sameSite=lax`; 30-day expiry; validated server-side every request; invalidated on logout.
- **At-rest encryption** (AES-256-GCM, key from `ROAMARR_SECRET`): travel-document `number`, fare-provider `api_key`, SMTP `smtp_pass`. App refuses to boot without `ROAMARR_SECRET`.
- **Cards:** store `last4` + `network` only — never the full PAN. This removes PCI scope by construction.
- **Authorization:** centralized in `sharing.ts`; public projection whitelists fields (§7).
- **Registration:** disabled by default; only the first-run admin exists until an admin enables it.

## 9. Testing (v0.1)

Vitest unit tests on the security- and logic-critical seams only (ponytail: one runnable check per non-trivial path, no E2E yet):

1. **`sharing.canView/canEdit`** — full matrix: owner, shared-user, group-member, public-token, stranger. (Security-critical.)
2. **`reminders.computeFireAt` + upsert** — correct fire times, idempotent re-upsert, past-time → immediate `sent`, cancel on delete.
3. **`auth`** — hash/verify round-trip; session validate + expiry.
4. **`crypto`** — encrypt/decrypt round-trip; tampered blob rejected.
5. **`fareproviders.registry`** — resolves the stub; `runFareChecks` skips disabled/paused.

## 10. Config & deployment

| Env var | Default | Notes |
|---|---|---|
| `ROAMARR_SECRET` | — (**required**) | Key material for at-rest encryption; boot fails if unset. |
| `DATABASE_PATH` | `/data/roamarr.db` | SQLite file on the mounted volume. |
| `PORT` | `3000` | adapter-node listen port. |
| `ORIGIN` | — | Public origin, required by SvelteKit for form actions behind a reverse proxy. |
| `SMTP_*` | — | Optional fallback; SMTP is normally set in `/settings`. |

Multi-stage Dockerfile: build SvelteKit, run `node build` with `/data` volume. Migrations apply on boot; first boot has `setup_complete=false` → `/setup`.

---

## 11. Out of scope for v0.1 (recap)

Email auto-import (later phase), real fare/seat providers (only stub), non-SMTP channels, extra segment types, configurable reminder leads, global benefit library, SSO/2FA, .ics sync. Each is a later spec.
